# CRM 미구현 기능 점검 — NAS 백엔드 연동 범위 설계

> 작성: 2026-07-14 · 상태: **분석만 완료, 구현 착수 전 (오너 검토 대기)**
> 제약: 기존 Supabase URL·키·RLS·CRM 기능 **일절 변경 없음**. 이 문서는 코드를 1줄도 바꾸지 않은 현황 분석이다.
> 관련: `NAS_테스트_구성_설계.md` (테스트 DB 스택 — 본 문서의 게이트웨이와는 **별개 컨테이너**)

## 총평

5개 기능 모두 프론트엔드에 **연동 지점(stub)이 이미 준비**되어 있고, 안 되는 이유는 공통적으로 "**비밀키·상시 실행·대용량 저장을 클라이언트가 가질 수 없어서**"다. 즉 NAS에 **작은 API 서버 1개(단일 Node 컨테이너)** 를 올리면 1~4번이 그 위에서 순차 해결된다. 5번(네이버)만 기술이 아니라 **제휴 절차**가 게이트다.

단, 조사 결과 "stub 내부만 교체하면 된다"는 기존 주석이 **정확하지 않은 곳이 2군데** 있다(아래 ⚠️ 표시): ① 발송 페이로드에 수신자 번호가 없음, ② photoStore가 동기 API라 네트워크 전환 시 호출부 비동기화 필요.

## 1. SMS·카카오 실제 발송 게이트웨이 (우선순위 1)

### 현재 코드 위치
| 파일 | 역할 |
|---|---|
| `src/lib/messagingGateway.ts` | 연동 지점. localStorage의 게이트웨이 URL/키가 있으면 fetch, 없으면 `pending` 반환 |
| `src/pages/Messaging/index.tsx:592` | 단체 발송 (pending 시 발송이력에 failed로 정직하게 기록) |
| `src/pages/Customers/index.tsx:324` | 상담 결과 카카오 전송 |
| `src/pages/Settings/index.tsx:54` | SMS 테스트 발송 버튼 |

### 실제로 안 되는 이유
1. 게이트웨이 URL/키가 미설정 → 모든 호출이 `{ pending: true, reason: '게이트웨이 미연동' }`.
2. 발송사(SMS=엔포, 카카오=알림톡) API 키를 클라이언트에 둘 수 없음 → 서버 필수.
3. **⚠️ 구조적 공백**: `SendPayload`가 `recipients: number`(인원수)만 담고 **전화번호 목록이 없다**. 게이트웨이를 붙여도 "누구에게" 보낼지 전달이 안 됨. 카카오 상담 전송도 마찬가지(고객 번호 미포함).
4. 카카오 알림톡은 **템플릿 사전 승인**(카카오 비즈메시지) 절차 필요 — 자유 문안 발송 불가. 채널 미가입 고객에겐 SMS 대체발송 설계 필요.

### NAS 백엔드 필요 범위
- `POST /api/messages/send` — 수신자 목록+문안 수신 → 엔포 SMS/알림톡 API 호출 → 건별 성공/실패 반환.
- 발송사 API 키는 NAS 서버 `.env`에만 보관. 요청 인증은 발급 토큰(Bearer) — 프론트 Settings에 저장하는 기존 `crm_sms_gateway_key` 구조 재사용 가능.
- 발송 로그 서버 기록(중복발송 방지·건수 정산 근거).
- **한도 제어**: 1회 요청당 최대 수신자 수·시간당 발송량 상한 (오발송 폭주 방지 — 원칙 4).

### 구현 시 프론트 변경 (비코어, 검토 후)
- `SendPayload`에 `phones: string[]` 추가 + 3개 호출부에서 실제 번호 전달. 코어 `store.ts` 무변경.

## 2. 예약 발송·재방문 자동 알림 (우선순위 2)

### 현재 코드 위치
| 파일 | 역할 |
|---|---|
| `src/pages/Messaging/index.tsx:800` | "예약 발송" 버튼 — `alert('NAS 서버 연동 후 지원')`로 종료되는 죽은 버튼 |
| `src/lib/reminderEngine.ts` | 재방문 대상 산출 로직(권장일/주기 기반) — **로직은 완성**, 클라이언트에서만 동작 |
| `src/components/RevisitReminderCard.tsx` | 대시보드 카드 — 대상 표시+메시지 초안 **수동 복사**까지만 |

### 실제로 안 되는 이유
- 예약 발송/자동 알림은 **앱이 꺼져 있어도 정해진 시각에 실행**돼야 하는데, CRM은 Electron/브라우저 앱이라 상시 실행 보장이 없음. 클라이언트에는 스케줄러를 둘 수 없는 구조적 한계.
- 자동 발송의 중복 방지(같은 고객에게 매일 재발송 금지)는 서버 측 발송 이력이 있어야 가능.

### NAS 백엔드 필요 범위 (기능 1에 의존)
- **스케줄러**(node-cron 등): ① 예약 발송 큐 폴링 → 시각 도달 시 기능 1 게이트웨이로 발송, ② 매일 정해진 시각에 재방문 대상 산출 → 자동 발송.
- 서버 저장소: `scheduled_messages` 큐 + 자동발송 이력. → **기존 클라우드 Supabase에 신규 테이블 추가가 자연스러우나 "기존 Supabase 변경 금지" 제약과 충돌**. 대안 2안을 오너 결정 사항으로 남김:
  - A안: NAS 로컬 SQLite/파일 큐 (클라우드 무변경, 단 데이터 이원화)
  - B안: 클라우드 Supabase에 **신규 테이블만 추가** (기존 테이블·RLS 무변경 — "변경"의 허용 범위 확인 필요)
- 재방문 대상 산출을 서버에서 하려면 고객·예약·시술 데이터 **읽기**가 필요 → NAS 서버가 기존 Supabase를 읽기 전용으로 조회(키는 서버 보관). reminderEngine.ts 로직을 서버로 이식.

### 구현 시 프론트 변경
- 예약 발송 버튼: alert 제거 → 발송 예약 API 호출 + 예약 목록/취소 UI.
- Settings에 자동 알림 on/off·발송 시각 설정.

## 3. 관리자 계정 생성 API (우선순위 3)

### 현재 코드 위치
| 파일 | 역할 |
|---|---|
| `src/lib/adminApi.ts` | 연동 지점 — `createBranchAdmin()`이 무조건 `pending` 반환 (교체 예시 주석 포함) |
| `src/pages/Admin/Branches.tsx:97` | 지점 생성 시 호출. 지점 레코드는 Supabase에 정상 생성되고 **계정만 pending 표시** |

### 실제로 안 되는 이유
- 계정 생성(`auth.admin.createUser`)은 `service_role` 키 필요 — 프론트에 두면 Critical 보안 결함이라 2026-07-10 감사에서 **의도적으로 제거**한 것. 서버 없이는 원천 불가.

### NAS 백엔드 필요 범위
- `POST /api/admin/create-user` — service_role 키(NAS `.env` 보관)로 supabase-js admin client 호출: auth 유저 생성 + `user_profiles` 연결 + 임시 비밀번호/초대 메일 처리 방식 결정.
- **기존 Supabase URL·키·RLS 무변경** — 이미 존재하는 service_role 키를 서버가 사용하는 것뿐(신규 발급·설정 변경 없음).
- 인증: 관리자 전용 토큰 + 요청자 검증(호출자가 본사 admin인지). 감사 로그.
- 5개 중 **구현 규모 최소** — 게이트웨이 서버 뼈대 검증용 첫 엔드포인트로 적합.

### 구현 시 프론트 변경
- `adminApi.ts`의 pending 블록 → fetch 교체 (주석의 예시 코드 그대로). 호출부 무변경.

## 4. 시술 사진의 NAS 저장·조회 (우선순위 4)

### 현재 코드 위치
| 파일 | 역할 |
|---|---|
| `src/lib/photoStore.ts` | 저장 계층 — localStorage에 base64 (기기별). 리사이즈/압축 포함 |
| `src/pages/Treatments/index.tsx:219` | 시술 전/후 사진 업로드·표시 |
| `src/components/TimelapseViewer.tsx` | Before/After 타임랩스 (사진을 날짜순 재생) |
| `src/pages/Customers/index.tsx` | `resizeImageFile` 재사용 |

### 실제로 안 되는 이유 (정확히는 "제한적으로만 됨")
- localStorage 한계: **기기별 격리**(다른 PC에서 안 보임), 용량 ~5MB(사진 수십 장이면 QuotaExceeded), 브라우저 데이터 삭제 시 유실.
- **⚠️ 기존 주석("저장 계층만 교체하면 됨")은 부정확**: `getPhotos`/`setPhotos`가 **동기 함수**다. NAS API로 바꾸면 비동기가 되므로 Treatments·TimelapseViewer 호출부의 비동기화 리팩터링이 필요 — 함수 내부 교체만으로는 안 됨.

### NAS 백엔드 필요 범위
- 파일 API: `POST /api/photos/:entityKey`(업로드) / `GET`(목록·조회) / `DELETE`.
- 저장 위치: `/volume1/CRM-STORAGE/photos/<shopId>/<entityKey>/` — CRM-STORAGE 활용(오너 지시 부합). CRM-BACKUP으로 일일 백업 포함.
- **고객 얼굴 사진 = 민감 PII**: 토큰 인증 필수, 직접 URL 노출 금지(서명 URL 또는 인증 프록시), 백업 암호화 검토. NAS는 외부 비공개(테스트 구성과 동일하게 루프백+터널 원칙)라 **원격 접근 방식이 선결 과제** — 매장 밖에서 사진 조회가 필요한지가 설계 분기점(오너 확인 필요).
- 마이그레이션: 기존 localStorage 사진 → NAS 일괄 업로드 유틸.

### 구현 시 프론트 변경
- photoStore 비동기 API 전환 + 호출부 2곳 리팩터링(비코어). 실패 시 localStorage 폴백 유지 권장(오프라인 대비).

### 지금 할 수 있는 것
- 솔루션 파트너 등록 **신청 절차만 병행 시작** (심사 기간이 길 수 있음 — 4번까지 구현하는 동안 대기열에 올려두는 게 효율적). 그 전까지는 현행 수동 입력(예약경로=네이버)이 공식 경로.

## 구현 순서 제안 (오너 검토용)

오너 우선순위(1→5)를 존중하되, 의존관계상 **공통 서버 뼈대**가 선행되어야 하고, 3번이 가장 작아 뼈대 검증에 적합하다:

```
0. NAS 게이트웨이 서버 뼈대 (Node 단일 컨테이너, 루프백 바인딩, 토큰 인증, 로깅)
   └─ NAS_테스트_구성_설계.md의 포트·비공개 원칙 그대로 적용
1. [P3] 관리자 계정 생성      ← 가장 작음, 뼈대 검증 겸용 (외부 발송사 의존 없음)
2. [P1] SMS·카카오 발송       ← 엔포 계정·알림톡 템플릿 승인 절차 병행 필요
3. [P2] 예약 발송+재방문 자동  ← 2의 게이트웨이에 스케줄러 추가 (큐 저장소 A/B안 결정 필요)
4. [P4] 사진 NAS 저장          ← 프론트 비동기 리팩터링 동반 (원격 접근 요건 확인 필요)
5. [P5] 네이버 동기화          ← 파트너 승인 대기 (신청만 지금 시작)
```

### 오너 결정 필요 사항 (구현 전)
1. **발송 큐 저장소**: NAS 로컬(A) vs 클라우드 Supabase 신규 테이블만 추가(B) — "기존 Supabase 변경 금지"의 허용 범위
2. **사진 원격 접근**: 매장 밖에서도 사진 조회가 필요한가? (필요하면 NAS 비공개 원칙과의 조율 필요)
3. **발송사 계약**: 엔포 SMS 계정·카카오 비즈메시지(알림톡) 채널·템플릿 승인 — 계약 주체와 일정
4. **네이버 파트너 신청**: 지금 신청 시작할지
